# Multi-Strategy Extraction Engine: Router Orchestration

The **Extraction Router** (`src/agents/router.py`) is the core orchestration component of the Phase 2 Extraction Engine. It reads the `DocumentProfile` generated by the triage agent and orchestrates a page-by-page extraction process.

Its design fulfills strictly the requirement of a **Documented extraction router with confidence-based escalation and graceful degradation.**

## Core Features

### 1. Confidence-Based Escalation (Fallback Chain)
The system does not blindly apply expensive models to all pages. Instead, it natively utilizes a confidence-gated escalation chain evaluated dynamically per page:

- **Baseline Selection:** Based on the document's `OriginType` and `LayoutComplexity`, a baseline strategy is chosen (e.g., `FastTextExtractor` for single-column native digital documents).
- **Page-Level Evaluation:** As the document is processed page-by-page, the baseline extractor runs and generates a `confidence_score`.
- **Escalation Threshold:** The threshold (e.g., `0.65`) is defined entirely in `rubric/extraction_rules.yaml`. If the `confidence_score` falls below this threshold (e.g., due to a complex table on page 4), the router activates its escalation chain.
- **Lazy-Loaded Fallbacks:** The router yields subsequent extractors sequentially (`LayoutExtractor` via Docling, then `VisionExtractor` via VLM). Crucially, these heavier models are **lazy-loaded**—they only consume memory if escalation is actually required.

### 2. Graceful Degradation
To ensure maximum reliability in production batch-processing, the router employs protective error boundaries:

- **Exception Catching:** If any strategy crashes (e.g., PDF stream errors, API timeouts, memory bounds exceeded), the exception is caught and logged. 
- **Empty Page Continuity:** The router records an empty `ExtractedPage` with a confidence score of `0.0` and the failure reason, and gracefully moves on to the next page. It **never** brings down the entire document pipeline due to a single unreadable page.
- **Checkpointing:** Extraction progress is locally cached in checkpoints per page, so if the process itself is externally interrupted, progress is saved.

### 3. Structural Validation
Before finalizing a page's extraction, the `ExtractionValidator` analyzes the output:
- Detects hallucinations or abnormally short content limits compared to the original page properties.
- Checks table matrix consistency (e.g., row length vs. header column length mismatches).
- If validation fails, it issues a protective penalty to the `confidence_score`, which inherently triggers the escalation chain mentioned above without requiring manual intervention.

### 4. Dynamic Budget Guard
When escalating to the VLM (`VisionExtractor`), the router shares a stateful `BudgetGuard` entity. This ensures cumulative API cost *across all pages* within the document never exceeds the `max_usd_per_document` defined in the configuration. 

## Externalized Configuration
All thresholds guiding this logic (escalation triggers, budget limits, origin routing matrix) and **all domain classifications** are strictly maintained as variables in `rubric/extraction_rules.yaml`, ensuring 100% config-only domain onboarding.
