# The Document Intelligence Refinery: Domain Conceptualization and Architectural Decisions

## The Problem Space
Extracting structured intelligence from uncooperative enterprise documents is a historically expensive and error-prone undertaking. Traditional Optical Character Recognition (OCR) systems destroy semantic context (turning tables into garbled strings), while multimodal Large Language Models (VLMs) hallucinate wildly and incur unbounded computational costs when pointed naively at 200-page PDF reports. 

The primary business objective is to build a deterministic "funnel" that balances cost, speed, and structural fidelity—deploying high-cost tools exclusively when lower-cost heuristics fail.

## I. Extraction Strategy Tree 

The Refinery employs a three-tiered escalation protocol. It applies the "minimum viable cost" to every page, escalating only when confidence metrics predict an extraction failure.

### Strategy A: The Fast Text Extractor (`pdfplumber`)
* **Cost:** ~$0.001 per page.
* **Speed:** 10-50 pages per second.
* **Use Case:** Clean, native-digital documents (e.g., modern Word-to-PDF reports) that consist primarily of continuous narrative text.
* **Mechanism:** Iterates physical PDF bounding boxes, clustering tokens into lines based on spatial tolerances. It evaluates the raw character density and image-ratio of the page to generate a 0.0-1.0 confidence score.
* **Failure Modes:** Utterly fails on scanned documents, and destroys the row/col relationships within complex financial tables.

### Strategy B: The Layout-Aware Engine (`Docling`)
* **Cost:** ~$0.02 - $0.05 per page (amortized compute).
* **Speed:** 1-5 seconds per page.
* **Use Case:** Documents containing dense structural elements like multi-column layouts, nested lists, and explicit financial tables. 
* **Mechanism:** Employs computer vision object detection and `TableFormer` ML architectures to identify layout elements before applying OCR. 
* **Failure Modes:** Memory intensive. Prone to Out-of-Memory (OOM) fatal errors on heavily graphical or anomalous pages. *Architectural Solution: Extractor runs in highly-isolated Subprocesses to contain memory leaks without crashing the main pipeline.*

### Strategy C: The Vision-Augmented VLM (`OpenAI/Gemini Multimodal`)
* **Cost:** $0.05 - $0.20+ per page (Strictly capped).
* **Speed:** 5-15 seconds per page (API latency bound).
* **Use Case:** Highly uncooperative documents, degraded historical scans, and dense infographics where OCR natively fails.
* **Mechanism:** Renders the PDF page to a base64 PNG, passing it to an LLM with strict JSON Schema directives to parse exact structural components.
* **Failure Modes:** Hallucinations and unbounded API bankruptcy. *Architectural Solution: A `BudgetGuard` state object is passed horizontally across page extractions, tracking cumulative USD spend and immediately throwing an exception if the per-document budget limit is breached.*

## II. Semantic Chunking & Constitution

Extracting text is only half the battle. If a 10,000-token financial report is sliced arbitrarily to fit within an LLM's context window, context is severed. A table row separated from its header row is useless data. The Refinery enforces five "Constitutional" rules for generating Logical Document Units (LDUs):

1. **Tables Cannot Be Split:** A `TableBlock` must remain a single, contiguous LDU regardless of max-token constraints.
2. **Figures Bind to Captions:** Text strings immediately below a bounding box flagged as an image/figure must be embedded together.
3. **Numbered Lists Stay Grouped:** Serialized itemization (`1., 2., 3.`) must not be severed across chunk boundaries.
4. **Headers Propagate Downward:** If a chunk contains an identifying section header, that context must be injected as metadata into all subsequent LDUs until a new header supersedes it.
5. **Spatial Order Preservation:** Chunks are ordered explicitly by physical `(top, left)` coordinate boundaries, not just internal datastream arrays.

These rules guarantee that the downstream `PageIndex` tree and ChromaDB VectorStore ingest coherent, stand-alone semantic concepts rather than broken sentence fragments.

## III. The Audit & Query Interface

The final layer is the Query Agent, an intelligence supervisor built on `LangGraph`. It possesses three distinct tool-arms to retrieve information:
1. **Semantic Vector Search:** For open-ended natural language questions spanning the corpus.
2. **PageIndex Navigation:** For jumping directly to specific chapters using the hierarchically LLM-summarized graph.
3. **SQLite Fact Table:** For deterministic numeric queries (e.g., "What was the total revenue in Year 3?") leveraging rigid structured extraction data unaffected by standard RAG hallucinations. 

**The Provenance Imperative:**
Every response generated by the Agent must carry an unbroken trail of evidence. By explicitly passing the Document ID, Page Number, and Bounding Box of the derived LDUs into the Agent State, the final VLM output acts less like an oracle, and more like a verifiable legal paralegal pointing directly to the source text.
